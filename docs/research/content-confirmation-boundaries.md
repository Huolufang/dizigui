# 内容确认的来源优先级与人机边界

## 结论

现有四份资料足以自动生成审核清单、异文清单、候选映射和完整性报告，但不足以把18门课程或90个原文单元自动标记为“已确认”。根据已确认基线，自动化只能证明“资料存在、结构完整、引用可解析、候选关系可复查”；规范原文裁决、注释译文是否对应、讲稿是否直接相关、释义差异是否影响理解、生僻字实际读音，以及课程最终确认，均必须由产品所有者逐项确认并留痕。

因此，“108项内容审核”宜理解为 **18门课程审核 + 90个原文单元审核**，而不是108个可以一次批量通过的同质检查。课程和单元的自动检查可以批量运行，人工结论不能用抽样或默认值代替；PRD明确要求内容验收不得抽样替代逐课确认（`docs/PRD-v2.md` 第5.4节，第581—601行）。

## 核查范围与方法

本报告只使用仓库内以下一手输入，不以外部网页替代仓库中已经冻结或待裁决的资料：

- `docs/PRD-v2.md`：需求与验收规则；
- `content/raw/弟子规-题解注释译文.md`：公开学习底稿；
- `miniprogram/data/legacy-course.js`：旧课程与90个原文单元的对照输入；
- `miniprogram/data/lectures.js`：现有40讲文字稿快照。

机械核查采用字段/区块计数、ID去重、讲次连续性、去除常见标点后的原文比对，以及讲稿原句检索。这里的“机械确认”仅表示可重复的数据事实，不表示内容学术正确或产品已经验收。

## 1. 规则来源：PRD优先于所有内容数据

`docs/PRD-v2.md` 已标记为“已确认基线”，并声明是课程重编、数据改造和验收的唯一需求依据（第1—7行）。它还明确 v1.1.0 只是功能原型和内容输入，不保留旧结构的权威性（第13—22行）。因此，四份文件发生用途或流程冲突时，应先执行 PRD，而不能用旧数据的现状反推验收规则。

PRD规定三类内容必须独立保存并可追溯（第2.1节，第91—109行）：

1. 《弟子规》原文是固定学习对象。公开学习底稿中的【原文】区块只是规范原文候选，旧课程的原文单元只是对照；差异必须进入异文清单，由产品所有者裁决。裁决后还需生成受控规范原文，并由产品所有者逐句确认、登记版本与校验值（第95—99行）。
2. `lectures.js` 所承载的40讲是课程解释和生活启示的主要来源，但当前缺少原始出版物、网页或整理版本信息，不能虚构其版本权威性（第100—103行）。
3. 公开学习底稿只用于补充讲稿未覆盖的字词注释和基础译文，必须保留其“公开学习资料整理版”身份，不得冒充权威定本或蔡老师观点，也不得静默改写（第104—107行）。

解释内容的优先级也由 PRD 固定：原文字句不被讲解改变；蔡老师讲稿为主要解释；公开学习底稿作补充；课程标题、导语和通用反思问题只能承担导航作用，不得新增解释性观点（第2.2节，第111—127行）。当讲稿和底稿存在不同理解时，主课程采用讲稿解释，但必须同时保存两种来源及差异，不得生成折中解释（第120—127行）。

## 2. 四个来源各自能证明什么

### 2.1 `docs/PRD-v2.md`

它能证明审核规则、责任归属和完成条件，不能代替对具体内容作结论。

- 课程与讲稿必须是段落级、多对多映射；旧式“第5—8讲”范围不能成为正式映射（第2.4节，第143—162行）。
- AI可以生成映射候选，但每个映射必须打开原讲稿段落，由产品所有者人工确认（第160行）。
- 单课审核必须覆盖原文字句与分组、读音、底稿位置、讲稿直接相关性、段落定位、阅读量、释义差异、问题来源和产品扩写等项目（第2.5节，第164—190行）。
- 内容必须同时通过人工审核和自动检查；自动检查负责 ID、引用、覆盖、字数、状态和例外记录等可判定事实（第2.6节，第192—211行）。
- 数据模型要求来源、原文单元、底稿片段、讲次、讲稿段落、展示片段、课程、映射、反思问题和内容差异分别保存（第4.2节，第395—414行）。
- 最终仍要求产品所有者逐项确认课程内容和最终版本（第5.5节，第603—618行）。

### 2.2 `content/raw/弟子规-题解注释译文.md`

该文件是完整的公开学习底稿输入，但它自己已经限制了权威性：开头说明其主要据 ruiwen.com 等公开资源和通用字词注解汇编，并明确不是中华书局第三版原版解读（第1—5行）；末尾再次说明题解与译文综合公开网页，注释为学习用途补充的通用字词训释（第608—612行）。

机械统计得到：

- 八个篇章完整出现：总叙、入则孝、出则悌、谨、信、泛爱众、亲仁、余力学文（章节起点见第9、26、126、198、306、402、498、528行）；
- 共有88个【原文】、88个【注释】和88个【译文】区块；
- 有42处括号拼音形式，可自动提取为“已有读音候选”，例如“悌（tì）”（第18行）、“凊（qìng）”和“省（xǐng）”（第42—43行）。

88个底稿区块不等于90个原文单元。去标点比较显示，其中总叙第13行对应旧单元 `DZG-ZX-001` 与 `DZG-ZX-002` 两个单元，入则孝第30行对应 `DZG-RX-001` 与 `DZG-RX-002` 两个单元。自动化可以生成这种拆分候选，却不能据此宣称注释和译文已经逐单元准确映射；拆分后某条注释或译文属于一个还是两个单元，仍需核对语义范围。

### 2.3 `miniprogram/data/legacy-course.js`

机械统计显示该文件有21个天次，其中18个 `lesson`、2个 `checkpoint`、1个 `graduation`，共90个不重复的原文单元 ID。每个旧课程记录含 `day/type/stage/chapter/title/estimate/units/focus/boundary/question/cai` 等字段（典型结构见第1—24行），90个单元及其稳定 ID 可作为规范原文编排的对照输入。

但此文件不能证明18门正式课程已经确认：

- PRD已否定固定21天结构的权威性，并要求按完整语义、讲稿密度和学习负荷重编（`docs/PRD-v2.md` 第2.3节，第129—141行）。
- `cai.lecture` 只是“第5—8讲”等范围字符串，`points` 和 `case` 也是压缩后的编辑文本，没有讲次 ID、段落 ID或逐单元映射（`legacy-course.js` 第20—23行）。
- `focus`、`boundary`、`question` 含产品编写的现代判断，例如第17—23、40—46、461—467行；它们可以作为候选选题，但不能作为蔡老师讲稿或公开底稿的原文引用，也不能默认通过“不存在产品自行扩写”的验收。
- 检测和结业记录没有原文单元（第143—159、307—323、470—487行），进一步说明“21天”与“18门内容课程”不是同一个审核对象。

### 2.4 `miniprogram/data/lectures.js`

文件头声明它由脚本从用户提供的40讲文字稿机械生成，正文不作摘要或改写（第1—3行）。机械解析可确认：讲次编号1—40连续存在，共1477个现有段落；第1讲字段结构为 `number/title/paragraphs`（第4—8行），第40讲从第1674—1677行开始，查询接口按讲次编号返回数据（第1723—1731行）。这足以确认“相对于本快照”的讲次和正文数组完整读取。

它不能证明外部版本正确，也还不能直接满足正式映射的数据要求：文件没有来源 ID、外部版本、文件校验值、校订记录或稳定段落 ID；数组位置只能用于生成待冻结的候选段落 ID。PRD也明确，40讲完整性验收只针对受控快照，不声称与某一出版或网络版本完全一致（`docs/PRD-v2.md` 第5.4节，第587—590行；附录A，第631—638行）。

## 3. 建议的来源优先级

| 场景 | 第一优先级 | 第二优先级/用途 | 不得自动采用 |
| --- | --- | --- | --- |
| 审核规则和完成条件 | 已确认基线 `docs/PRD-v2.md` | 无 | 用旧数据现状覆盖 PRD |
| 规范原文 | 裁决后的受控规范原文（当前四文件中尚不存在） | 底稿【原文】作候选；旧90单元作对照 | 任一候选自动胜出 |
| 解释与生活启示 | `lectures.js` 中可定位的蔡老师原段落 | 底稿仅补讲稿未覆盖的基础注释/译文 | `legacy-course.js` 的 `focus/boundary/cai.points/case` 冒充来源正文 |
| 基础注释与译文 | 公开学习底稿的原始片段 | 逐单元引用并保留来源 | 静默改写或宣传为权威定本 |
| 释义冲突 | 课程解释采用讲稿，同时保留底稿、讲稿和差异记录 | 产品所有者判断是否影响理解 | 自动折中、删去一方或默认“无差异” |
| 生僻字读音 | 产品所有者选定并登记的权威字典/正式注释来源 | 底稿42处拼音作候选线索 | 仅凭整理版拼音宣称全部正确完整 |
| 课程结构与问题 | 按 PRD 重编并逐课确认 | 旧18个 lesson 仅作候选 | 继承旧21天结构即视为正式课程 |

其中“规范原文第一优先级”目前是一个待产出的受控文件，而不是四份输入中的任一现存文件。PRD指定其最终路径为 `content/raw/dizigui-original.md`，并要求登记版本和校验值（附录A，第622—638行）。在该文件完成逐句确认前，90个单元只能处于候选或待审核状态。

## 4. 可以机械确认的项目

以下项目可由脚本确定并输出证据，不需要产品所有者逐条做重复劳动：

1. **结构与计数**：18个旧 `lesson`、90个唯一原文单元、88组底稿区块、40个连续讲次、1477个现有讲稿段落。
2. **ID与引用完整性**：来源、课程、原文单元、讲次、段落和映射 ID 是否存在、唯一；引用是否失效；讲次是否缺号。
3. **原文候选比对**：去除指定标点和空格后做精确比较，报告遗漏、重复、拆分/合并关系和所有异文候选。
4. **底稿候选映射**：按篇章和相邻【原文—注释—译文】区块，生成到90个单元的候选引用范围。
5. **讲稿候选检索**：查找原句或关键词命中的讲次和段落，生成可打开原文的候选映射，不能生成最终映射结论。
6. **量化校验**：讲稿重点字数、段落重复复用、原文遗漏/重复、未找到对应讲稿的显式记录。
7. **流程完整性**：每个单元是否填写差异状态、读音候选、确认人、确认日期；每门课程是否填写来源、试读时长、审核状态和例外记录。
8. **快照完整性**：第1—40讲相对于 `lectures.js` 当前快照是否全部进入产品、是否发生正文截断。结论必须写“相对于该快照”，不能扩张为外部版本一致。

机械检查结果只能产生“通过、告警或候选”，不能替产品所有者写入语义性“已确认”。

## 5. 必须由产品所有者人工确认的项目

### 5.1 90个原文单元逐项确认

每个单元至少需要确认：

- 采用的规范字词、篇章名、标点和单元边界；
- 底稿注释和译文是否确实对应本单元，拆分后是否存在跨单元引用、漏义或多引；
- 讲稿候选段落是否与本原文直接相关，而非仅主题相似；
- 讲稿与底稿的差异状态是“无差异、有差异、不适用”中的哪一种；若有差异，关键字义、主旨或行为结论是否影响学习者理解；
- 多音字或生僻字在该句中的实际读音，以及采用的核对来源。

PRD规定“未比较”阻断课程确认，最终差异判断由产品所有者确认并留痕（第127行）；不能批量把未审核单元预填为“无差异”。

### 5.2 18门课程逐课确认

每门课程至少需要确认：

- 原文组合是否保持篇章顺序和完整语义，课程负荷是否合适；
- 讲稿重点是否真的服务本课，哪些段落是必读、哪些只供深入阅读；
- 每个正式映射能否打开具体讲次和原始段落；
- 讲稿必读字数和完整五步流程试读时长是否符合范围，越界是否批准例外；
- 标题、导语、反思问题是否中性且来源合规；
- 是否混入旧 `focus/boundary/cai.points/case` 等无原文引用的现代解释；
- 单元的异文、读音、底稿和差异审核是否全部完成，之后才可把课程状态改为“已确认”。

PRD明确未确认课程只可预览，不得作为正式学习课程打包（第188—190行）。

## 6. 已发现的必须裁决样例

机械比对已经发现两处不是标点差异的原文异文，足以证明不能批量默认采用任一文件：

| 单元 | 公开学习底稿 | 旧90单元 | 讲稿快照提供的线索 | 结论 |
| --- | --- | --- | --- | --- |
| `DZG-CD-010` | “**进**必趋，退必迟。问起对，视勿移。”（底稿第184行） | “**近**必趋，退必迟；问起对，视勿移。”（旧课程第131行） | 讲稿第876行出现“进必趋”，第892、895行又出现“近必趋” | 讲稿快照内部也不一致，必须由产品所有者依据选定版本裁决，不能用多数命中自动决定 |
| `DZG-JIN-017` | “借人物，及时还。**人借物，有勿悭**。”（底稿第298行） | “借人物，及时还；**后有急，借不难**。”（旧课程第224行） | 讲稿第1144、1170行采用“后有急，借不难” | 属于整句差异，必须进入异文清单并记录采用理由 |

另有“工夫/功夫”的来源差别：底稿原文和旧单元均写“工夫到”（底稿第556行；旧课程第433行），讲稿多处写“功夫到”（如第1695行）。这里讲稿是在引用原文还是口语转写、是否影响规范原文与展示搜索，都应记录为比较项；不能静默统一替换。

## 7. 对108项审核的实际安排建议

建议把工作拆成两道门，而不是直接逐页凭感觉确认：

1. **自动预审门**：先生成90单元审核表和18课程审核表，自动填入所有可证明字段、候选片段、字数、来源位置和告警；先解决 ID、遗漏、引用失效等数据问题。
2. **产品所有者确认门**：按90单元逐项裁决“原文—底稿—讲稿—差异—读音”，再按18课程逐课确认“组合—重点—负荷—问题—来源”。只有一门课程所含单元全部通过且课程自身清单通过，才把状态改为“已确认”。

推荐状态使用 PRD 已定义的“草稿 → 待审核 → 已确认”（第168—172行），并让每个结论记录确认人、日期、来源位置和备注。自动化应生成告警和待裁决项，不替产品所有者签署结论。

## 最终边界

自动化能够可靠回答：**数据是否齐、引用是否通、文本哪里不同、候选段落在哪里、必读量是否超限、审核字段是否填完。**

只有产品所有者能够回答并承担最终结论：**采用哪种规范原文、注释译文是否贴合、讲稿是否直接相关、差异是否影响理解、读音是否可信、课程是否可以正式发布。**

这一区分不是额外流程，而是 PRD 已确认的验收条件；若跳过人工裁决，即使小程序功能和自动检查全部通过，也仍不能视为 v2.0 内容完成（`docs/PRD-v2.md` 第5.5节，第603—618行）。
